1. 产品概述
1.1 定位
一个允许用户自定义多AI模型（本地/云端）参与辩论、创意孵化等回合制游戏的Web平台。每个AI席位可独立配置模型和“个人Skill”，全局可配置“剧本”（全局Skill）和裁判插件。支持游戏历史隔离、回放、上帝干预。

1.2 核心特性
3~6个AI席位，星空主题UI，席位点亮与AI图标展示。

每个席位独立：模型选择（Ollama/OpenAI/DeepSeek等），个人Skill（角色扮演/任务指令）。

全局剧本：结构化JSON，定义最大轮次、裁判插件、轮次控制插件、提示模板等。

回合制：按顺序发言 → 裁判打分 → 进入下一轮，直至满足结束条件。

历史隔离：每场游戏独立保存快照（包含所有配置、发言、技能），支持回放。

上帝干预：游戏进行中用户可插入指令，强制影响后续发言。

插件化裁判与轮次控制：可插拔，内置LLM裁判和最大轮次控制，支持自定义脚本。

2. 技术栈
前端：Next.js + TypeScript + TailwindCSS + Framer Motion（动画）+ React-DnD（拖拽）

后端：Next.js API Routes + Prisma + SQLite（开发）/ PostgreSQL（生产）

AI调用：统一ModelProvider抽象（支持OpenAI、Ollama、DeepSeek等）

插件系统：基于TypeScript接口的插件注册表 + 沙箱执行（isolated-vm）用于用户脚本

3. 数据库设计（Prisma Schema）
prisma
// 游戏对局（每次开始新游戏创建一条）
model GameSession {
  id             String         @id @default(cuid())
  name           String
  createdAt      DateTime       @default(now())
  endedAt        DateTime?
  globalRule     String         // 结构化JSON字符串（剧本元数据+规则）
  status         GameStatus     @default(WAITING)
  winnerBotId    String?        // 胜者GameBot的id
  maxRounds      Int            // 从globalRule中提取，冗余便于查询
  rounds         Round[]
  participants   GameBot[]
}

// 游戏中的Bot快照（记录当时配置）
model GameBot {
  id             String         @id @default(cuid())
  sessionId      String
  session        GameSession    @relation(fields: [sessionId], references: [id])
  name           String
  model          String         // "openai/gpt-4", "ollama/deepseek-r1:7b"
  basePrompt     String?        // 本色提示词
  order          Int            // 席位顺序0-5
  finalScore     Int            @default(0)
  skillSnapshots Json           // 安装的所有Skill的深拷贝 [{id, name, type, content}]
  messages       Message[]
}

// 游戏回合
model Round {
  id             String         @id @default(cuid())
  sessionId      String
  roundNumber    Int
  startedAt      DateTime       @default(now())
  finishedAt     DateTime?
  session        GameSession    @relation(fields: [sessionId], references: [id])
  messages       Message[]
}

// 每条发言消息
model Message {
  id             String         @id @default(cuid())
  roundId        String
  gameBotId      String
  content        String         @db.Text
  skillSnapshot  Json           // 发言时合并后的完整提示词快照（含全局+个人+上帝指令）
  createdAt      DateTime       @default(now())
  round          Round          @relation(fields: [roundId], references: [id])
  gameBot        GameBot        @relation(fields: [gameBotId], references: [id])
}

// 独立的Skill库（可复用）
model Skill {
  id          String     @id @default(cuid())
  name        String
  description String?
  type        SkillType  // ROLE_PLAY, TASK
  content     String     // 提示词内容
  author      String?
  version     String     @default("1.0.0")
  preview     String?    // 示例回复
  tags        String[]
  isPublic    Boolean    @default(true)
  downloads   Int        @default(0)
  createdAt   DateTime   @default(now())
  updatedAt   DateTime   @updatedAt
}

enum GameStatus { WAITING IN_PROGRESS FINISHED }
enum SkillType { ROLE_PLAY TASK }
4. 核心模块接口定义
4.1 ModelProvider（抽象所有AI调用）
typescript
interface ModelProvider {
  call(params: {
    model: string;          // "openai/gpt-4"
    messages: Array<{role: string, content: string}>;
    temperature?: number;
    maxTokens?: number;
    apiKey?: string;
    baseURL?: string;
  }): Promise<string>;
}
内置实现：OpenAIProvider, OllamaProvider, DeepSeekProvider。

4.2 PromptMerger（合并全局规则+个人技能+上帝指令）
输入：

globalRuleStruct：结构化剧本（含promptTemplate）

botSkillSnapshots：该Bot安装的技能列表（ROLE_PLAY和TASK）

godIntervention？：上帝干预文本

chatHistory：最近若干条消息

输出：最终的system prompt + user message数组。

合并规则：

如果存在ROLE_PLAY技能，其content作为system prompt主体；否则用bot.basePrompt。

将globalRule中的promptTemplate渲染（替换{topic}等变量）后追加到system prompt。

将每个TASK技能的content作为附加指令追加。

如果存在godIntervention，将其以最高优先级插入system prompt开头。

将历史消息格式化为user/assistant交替。

4.3 回合控制器（API: POST /api/game/{sessionId}/next-turn）
流程：

验证状态为IN_PROGRESS。

获取当前回合数，若已达到maxRounds则结束游戏。

创建新Round，按order顺序遍历GameBot。

对每个Bot：

调用PromptMerger构建消息。

通过ModelProvider调用模型。

保存Message，复制当前bot.skillSnapshots到message.skillSnapshot。

所有Bot发言完后，如有裁判插件，调用裁判的onRoundEnd获取分数并更新GameBot.finalScore。

标记Round完成，增加session.currentRound（冗余字段）。

调用轮次控制插件的shouldEnd判断是否结束。若结束，调用裁判插件的onGameEnd，更新status为FINISHED，记录winnerBotId。

4.4 裁判插件接口

内置实现：LLMJudge（调用指定模型评分，需解析JSON）。

5. UI/UX设计
5.1 主界面布局
背景：深色星空渐变，动态星星（CSS/Three.js）。

中央六个星球：环形排列，每个星球代表一个席位。

未配置：灰色半透明，点击弹出配置面板。

已配置：点亮（光晕+颜色），显示AI头像（可自动生成或上传）。

右侧侧边栏（可收起）：

游戏进行时自动展开，显示分层轮次对话框。

每个轮次为一个折叠卡片，展开后展示该轮每个Bot的发言及得分。

每个卡片旁边有一个“上帝干预”按钮（仅当前轮次未结束时可用）。

顶部控制栏：

“新游戏”按钮 → 打开配置向导。

当前游戏状态（等待/进行中/已结束）、当前回合数/总回合数。

“下一回合”按钮（手动模式）或“自动进行”开关。

5.2 游戏配置向导（三步）
剧本选择：从预设结构化剧本中选取，或新建自定义（填写名称、主题、maxRounds、裁判模型等）。

席位配置：为每个席位选择模型（下拉列表从环境变量+Ollama动态获取）、上传/选择个人Skill。

确认并开始：创建GameSession，跳转到游戏房间。

5.3 历史回放模式
从首页“历史对局”列表进入，界面与游戏房间相同，但所有控制按钮禁用，侧边栏只读，无法发送上帝指令。

支持按轮次快速跳转。

6. 历史隔离与快照机制
每次点击“开始新游戏”时，后端创建一个新的GameSession，并将当前配置深拷贝到GameBot表（包括技能快照）。

发言时，将PromptMerger最终生成的system prompt和user消息的组合作为skillSnapshot存到Message中（或者只存最终system prompt，便于回放时精确重现）。

裁判评分结果只影响GameBot.finalScore，不影响快照。

游戏结束后，所有数据独立，可以安全删除原Skill库中的技能而不影响已结束游戏的回放。

7. 上帝干预实现细节
前端在next-turn请求中附带可选字段godIntervention: string。

后端在PromptMerger处理时，若存在该字段，将其以最高优先级插入system prompt的开头，格式：[上帝指令]：{干预内容}\n\n。

该干预内容同时作为一条特殊消息保存到该轮消息列表中（role="god"，gameBotId为空），便于回放时展示。
