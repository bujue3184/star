> 开发环境：Node.js v24.15.0, Ollama 0.30.0, RTX 5060 8GB

## 三、核心功能清单

### 3.1 3D 星空场景
- 2500 颗粒子星空（闪烁动画）
- 6 大星球环形排列（AI 形象贴图 + 光晕）
- 中心恒星（裁判席，脉动 + 日冕射线）
- OrbitControls 视角控制（自动旋转 + 拖拽）
- Raycaster 点击交互（区分恒星/星球）

### 3.2 AI 模型支持（5 家厂商）
| 前缀 | 厂商 | 类型 |
|------|------|------|
| `ollama/*` | Ollama 本地模型 | deepseek-r1, qwen, gemma 等 |
| `openai/*` | OpenAI | GPT-4o, GPT-4o-mini |
| `deepseek/*` | DeepSeek | deepseek-chat, deepseek-reasoner |
| `volcengine/*` | 火山引擎/豆包 | doubao-pro, doubao-lite |
| `dashscope/*` | 阿里云/千问 | qwen-plus, qwen-max, qwen-turbo |

### 3.3 流式 SSE 输出
- 逐 token 推送，前端实时显示
- 每个 bot 发言时对应星球高亮
- 裁判建议 + 玩家确认结束机制
- 游戏结束后 V4 自动生成总结

### 3.4 V4 导演统筹（最新特性）
- 每轮由 DeepSeek V4 决定谁发言、说什么
- 输出格式：`PHASE|阶段名` + `SPEAK|角色名|指令`
- 可指定 1-3 人发言，不必全员输出
- V4 认为讨论完成时可提前结束

### 3.5 上帝干预
- 上帝干预输入框（每轮可插入指令）
- 手动结束游戏按钮（按分数判定胜者）
- 裁判建议弹窗（确认/拒绝）

## 四、开发过程中解决的问题

### 4.1 显存管理（RTX 5060 8GB）
- **问题**：3 个本地模型同时加载 → 13GB > 8GB → OOM
- **解决**：
  - `keep_alive: "0"` 用完立即释放显存
  - 模型切换间隔 3 秒
  - 单次 Ollama 调用超时 120s → 300s

### 4.2 思维链剥离
- **问题**：deepseek-r1 的 `...` 标签、qwen3.6/gemma4 的 `Here's a thinking process:` 污染输出
- **解决**：
  - deepseek-r1：流式检测 `<think>`/`...` 标签，标签内跳过
  - qwen3.6/gemma4：检测 `thinking` 字段，优先取 `content`
  - 模块级共享状态 → 每路流独立检测器

### 4.3 星球闪烁错位
- **问题**：豆包发言时 GPT 星球闪烁
- **解决**：API 路由 `order: bot.order ?? index` 保留前端传入的席位索引

### 4.4 SSE 流控制器崩溃
- **问题**：浏览器断连后 `controller.enqueue()` 抛出 "Controller is already closed"
- **解决**：`safeEnqueue()` 捕获并忽略断连错误，其余 bot 继续执行

### 4.5 身份混淆
- **问题**：模型看到历史消息中的 `【xxx 发言】` 后模仿该身份
- **解决**：
  - 每次发言前重申身份 `【千问】，现在轮到你发言了。请记住你的设定：...`
  - 系统消息不再包含在 chatHistory 中
  - 所有默认 basePrompt 去掉"辩手"字眼

### 4.6 裁判评分系统
- **问题**：独立评分 → 全员趋同（7-9 分），无区分度
- **解决**：改为横向对比评分 + 强制排名
- **最终**：取消打分机制，裁判只做进程裁定 + 统筹规划

## 五、已知问题 / 待优化

### 5.1 模型能力差异
- deepseek-r1:8b（5.2GB）表现最好，推理架构优势明显
- qwen3.6:latest（23GB）太大，8GB 显存放不下，CPU 推理极慢
- gemma4:latest（9.6GB）同样超显存，输出不稳定
- gemma3:4b（3.3GB）太小，复杂任务容易崩
- **建议**：席位尽量只用 deepseek-r1:8b / qwen:7b / gemma3:4b

### 5.2 英文输出污染
- qwen3.6 和 gemma4 输出英文思维链，污染其他模型的上下文
- 思维链剥离已修复，但仍需验证

### 5.3 重复轮次
- 极端情况下两个请求同时到达会创建同号轮次
- 后续可加 Prisma 事务锁

### 5.4 裁判总结
- `/api/game/:id/end` 现在已支持游戏结束后生成总结
- 但需要用户手动点击「确认结束」按钮触发

## 六、API 接口清单

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/game` | 游戏列表 |
| `POST` | `/api/game` | 创建游戏 |
| `GET` | `/api/game/:id` | 游戏详情（含轮次消息） |
| `DELETE` | `/api/game/:id` | 删除游戏 |
| `POST` | `/api/game/:id/start` | 开始游戏 |
| `POST` | `/api/game/:id/turn` | 非流式下一回合 |
| `POST` | `/api/game/:id/turn/stream` | **流式**下一回合（SSE） |
| `POST` | `/api/game/:id/end` | 结束游戏 + 生成总结 |
| `GET` | `/api/models` | 探测可用模型列表 |
| `POST` | `/api/models/test` | 测试模型连通性 |
| `GET/POST` | `/api/skills` | 技能库 CRUD |
| `GET/PUT/DELETE` | `/api/skills/:id` | 单个技能 CRUD |
